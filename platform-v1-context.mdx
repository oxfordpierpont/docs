## 1) Program Boundary Addendum
### 1.1 Phase Boundaries
1. `Shell` phase delivers only control-plane infrastructure.
2. `Modules` phase delivers first-party modules and imported module runtime behavior.
3. `Capabilities` phase delivers capability graph, composition, and advanced interoperability tooling.

### 1.2 “Shell Done” Definition
1. Shell can run with zero installed modules.
2. Super, Agency, Business workflows are fully operable.
3. Module lifecycle exists as infrastructure only: import, validate, register, install, enable/disable, uninstall.
4. Layout Manager is integrated for shell surfaces only, per approved Layout Manager PRD.
5. Billing, tenancy, permissions, events, and gateway are enforceable and testable.

### 1.3 Module-Absent Operating Mode (Required)
1. Empty module registry must not break shell navigation.
2. Gateway must return deterministic `MODULE_NOT_ENABLED` or `MODULE_NOT_REGISTERED`.
3. Event bus must operate for shell-emitted events even with no module subscribers.
4. Admin module screens must gracefully show “No modules installed.”

8. `audit_logs`: write by system only, read by authorized admins and super.

## 5) Event + Gateway Operational Semantics
### 5.1 Event Delivery Rules
1. Ordering guarantee is per `(workspace_id, stream_key)` not global.
2. Idempotency key required for publisher retries.
3. Retry schedule: exponential backoff with max 5 attempts.
4. Dead-letter on terminal failure with replay endpoint.
5. Replay allowed only within workspace scope and authorized role.
6. Event schema validation occurs before persistence.

### 5.2 Gateway Rules
1. Resolve module target from `module_installations` + `module_registry`.
2. Enforce auth, workspace context, permission, billing, module-enabled checks before forward.
3. Forward signed context claims, not raw trust from client.
4. Timeout default 8s, circuit-break after threshold failures.
5. Prevent module impersonation via server-issued module identity binding.
6. Log every forwarded request with correlation id and enforcement outcome.

## 6) Billing Enforcement State Machine
States:
1. `active`
2. `past_due`
3. `grace`
4. `suspended`
5. `canceled`

Transitions:
1. `active -> past_due` on Stripe payment failure.
2. `past_due -> grace` when grace window starts.
3. `grace -> suspended` after grace expiry.
4. `past_due|grace|suspended -> active` on successful recovery payment.
5. `suspended -> canceled` on explicit cancel/termination.

Entitlement effects:
1. `active`: normal access.
2. `past_due`: warnings, no immediate lock.
3. `grace`: allow recovery actions, block new paid module activations.
4. `suspended`: block module usage; allow auth, billing recovery, account settings.
5. `canceled`: no module usage; reactivation required.

Webhook truth model:
1. Stripe webhook is source of truth for payment events.
2. Shell reconciliation job runs periodic consistency checks.
3. Entitlement updates are idempotent and audit-logged.

## 7) Measurable NFRs + Security Controls
### 7.1 SLO Targets (MVP)
1. Authenticated route guard decision p95: `< 200ms`.
2. Workspace switch completion p95: `< 500ms`.
3. Module route resolution p95: `< 300ms` (module absent path included).
4. Gateway forward overhead p95: `< 150ms` (excluding module runtime latency).
5. Layout publish/rollback command ack p95: `< 2s`.
6. Event enqueue latency p95: `< 250ms`.

### 7.2 Reliability Targets
1. No data-loss for committed layout versions.
2. Publish and rollback are idempotent by operation key.
3. Audit coverage for privileged actions: 100%.
4. Recovery from interrupted session restores last autosaved layout draft.

### 7.3 Security Controls
1. Deny-by-default permission evaluation.
2. Server-side enforcement independent of UI visibility.
3. Impersonation start/stop is explicit, scoped, and fully audited.
4. Gateway strips untrusted client headers and injects trusted claims.
5. Cross-workspace access denied unless explicitly super-scoped.

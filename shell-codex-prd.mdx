# aiConnected v2 Core Shell PRD (Implementation-Grade)
Version: `1.0`  
Date: `March 26, 2026`  
Status: `Authoritative MVP Build Specification`  
Owner: `aiConnected Platform`

## 0) Document Purpose and Rules of Interpretation
This document is the highest-priority source of truth for building the aiConnected v2 Core Shell MVP from a blank repository.

Interpretation rules:
1. If this document is explicit, implement exactly as written.
2. If this document is silent, choose the simplest approach that preserves future extensibility and strict shell/module boundaries.
3. If functionality could belong to shell or module, default to module unless explicitly marked shell-owned here.
4. Prioritize architectural correctness over UI polish, speed hacks, or convenience coupling.
5. Do not replace the required stack or core architecture decisions unless explicitly instructed.

Authoritative dependency:
1. The Layout Manager subsystem is governed by the Layout Manager PRD you finalized.
2. This shell PRD defines how Layout Manager is integrated, authorized, persisted, routed, and governed as a shell subsystem.

## 1) Product Identity
aiConnected v2 Core Shell is a white-label, multi-tenant SaaS control plane that owns:
1. Authentication and session lifecycle.
2. Tenant/workspace hierarchy.
3. Role and permission enforcement.
4. Shell routing and navigation.
5. Theme/branding inheritance.
6. Layout Manager integration.
7. Billing enforcement and access gating.
8. Module registry/import/activation.
9. Event bus and gateway infrastructure.
10. Shared platform entities and governance.

The shell is not:
1. A chat product.
2. A voice product.
3. A knowledge-base product.
4. A CRM product.
5. A marketplace product.
6. A workflow engine.
7. A business module.
8. An AI assistant runtime.

---

## 2) Product Goal
Deliver a durable shell foundation that allows agencies to run branded SaaS operations and add modules over time without shell rewrites.

Success outcomes:
1. Agency can onboard and operate platform as their own white-label product.
2. Agency can create and manage Business child workspaces.
3. Shell enforces permissions, billing, branding, and route access consistently.
4. Modules are installed/activated via contract-driven workflows, not hardcoded shell edits.
5. Platform remains stable while modules evolve independently.

---

## 3) Non-Negotiable Product Principles
1. Multi-tenant from day one.
2. White-label from day one.
3. Shared shadcn-based UI foundation from day one.
4. Centralized inheritance-based theming.
5. Layout Manager in MVP.
6. Module import/registration in MVP.
7. Dynamic module installability without hardcoded routes/pages.
8. Inter-module communication via events/contracts/gateway only.
9. No module business logic embedded in shell.
10. Core architecture must support unknown future modules.

Top-level user layers:
1. Super.
2. Agency.
3. Business.
4. Developer.
5. Personal.

MVP operationally supports:
1. Super full flow.
2. Agency full flow.
3. Business full flow.

MVP structural-only compatibility:
1. Developer.
2. Personal.

Core tenancy definitions:
1. Tenant boundary is represented by workspace ownership/control context.
2. Workspace is the runtime scope for permissions, routing, modules, and branding.
3. Agency workspaces are parent customer contexts.
4. Business workspaces are managed child contexts under agencies.
5. Membership links users to workspaces with role template + permission grants.
6. Super users can impersonate lower scopes with explicit audit traces.

## 9) Core Architecture
Shell layers:
1. Auth/session layer.
2. Tenant/workspace layer.
3. Permissions layer.
4. Theme/branding layer.
5. Layout layer.
6. Module registration layer.
7. Event layer.
8. Gateway layer.
9. Billing enforcement layer.

Shell/module boundary:
1. Module registers through manifest.
2. Shell validates, stores, and activates module contracts.
3. Shell exposes routes/navigation dynamically.
4. Shell checks permission and billing before module access.
5. Modules communicate through event/gateway contracts.
6. Modules cannot directly mutate other modules’ private data.
7. Shell never hosts module business workflows.

---

## 10) Shared Platform Data Model (Conceptual)
Required shell-owned entities:
1. `users`: identity records from auth domain.
2. `workspaces`: tenant runtime contexts.
3. `workspace_memberships`: user-workspace relation + role template.
4. `roles`: role template definitions.
5. `permissions`: permission catalog.
6. `role_permissions`: role-template grants.
7. `membership_permissions`: explicit overrides.
8. `themes`: workspace theme payloads.
9. `layout_definitions`: logical layout identities.
10. `layout_versions`: draft/published immutable snapshots.
11. `module_registry`: known module contract catalog.
12. `module_installations`: module enablement per workspace.
13. `billing_accounts`: billing owner/account linkage.
14. `subscriptions`: subscription state and enforceable status.
15. `events`: normalized event log.
16. `contacts`: shared cross-module contact entity.
17. `audit_logs`: privileged operations trail.

Ownership rules:
1. Shared entities above are shell-owned.
2. Modules can read allowed shared entities via contracts.
3. Modules own private tables for private domain data.
4. Cross-module coupling must use events/contracts, not direct table writes.

## 11) Authentication and Session Requirements
Required behaviors:
1. Sign-in and sign-out.
2. Session persistence.
3. Workspace resolution at login.
4. Workspace switching for multi-membership users.
5. Role-aware post-login destination.
6. Protected route handling.
7. Unauthorized/forbidden states.
8. Invitation-based onboarding.
9. Super impersonation entry/exit with explicit audit event.

Session context must always include:
1. `user_id`
2. `active_workspace_id`
3. `effective_role_template`
4. `effective_permissions`
5. `impersonation_state` if active

## 12) Multi-Tenant Workspace Requirements
Required behaviors:
1. Super can create/manage agency workspaces.
2. Agency can create/manage business child workspaces.
3. Users can belong to multiple workspaces with independent roles.
4. Data visibility is workspace-scoped by default.
5. Route and feature access resolve by active workspace context.
6. Theme inheritance cascades from agency to child business by default.
7. Module installation/enablement is workspace-scoped.
8. Parent agency can govern child workspace module availability in MVP.

Isolation rule:
1. No agency can access another agency’s business data.
2. No business workspace can access sibling business data.

## 13) UI Foundation and Theme Engine

1. shadcn/ui component baseline.
2. Tailwind utility layer.
3. Central theme token map consumed by shell surfaces.
4. White-label controls exposed in shell settings.
5. Consistent shell UI language across all shell routes.

Theme requirements:
1. Token categories include color, typography, radius, shadow, border, spacing, surfaces.
2. Agency-level token set can propagate to child workspaces.
3. Workspace-level override support is structurally possible in MVP.
4. Shell screens consume tokens from centralized theme service only.
5. No ad hoc page-level CSS divergence for branding logic.

## 14) Layout Manager Integration (Shell Perspective)
The full Layout Manager behavior is defined in the Layout Manager PRD. Shell must provide:

1. Entry points:
- In-context edit trigger on supported shell surfaces.
- Admin navigation to `Layout Manager > Modules` and `Layout Manager > Create New`.

2. Authorization:
- `layouts.edit` required for editing.
- Publish/rollback requires role+permission checks per Layout Manager PRD.

3. Persistence integration:
- `layout_definitions` and `layout_versions` stored in shell-owned data.
- Draft/published state and rollback metadata tracked in shell domain.

4. Rendering integration:
- Shell route renderers must resolve published layout snapshots for editable shell surfaces.

5. Governance:
- Save/Preview/Test/Publish/Rollback lifecycle enforced with audit logging.

6. Boundary:
- Layout Manager edits structure/composition only.
- Theme engine controls broad aesthetics.
- Backend/module logic remains outside layout authoring.

## 15) Module System Overview
Module shell contract:
1. Module is an isolated runtime unit registered to shell.
2. Shell recognizes module through validated manifest.
3. Shell dynamically exposes module route/nav entries.
4. Module can be enabled/disabled per workspace.
5. Module access is governed by permissions and billing state.
6. Module communication with others happens through event/gateway systems.
7. Shell never inlines module business behavior.

---

## 16) Module Manifest Contract
Required manifest fields:
1. `id`
2. `name`
3. `version`
4. `description`
5. `routes`
6. `sidebar` metadata
7. `required_permissions`
8. `capabilities`
9. `required_shared_entities`
10. `events_emitted`
11. `events_consumed`
12. `config_schema`
13. `runtime_target` (container/gateway target)

Validation rules:
1. Missing required fields reject import.
2. Invalid semantic version rejects import.
3. Duplicate module id with incompatible version policy rejects import.
4. Route collisions reject import unless explicitly namespaced/allowed.
5. Permission keys must exist in permission catalog.
6. Event names must match event naming contract.
7. Runtime target must be resolvable and marked healthy before activation.

Compatibility policy:
1. MVP supports additive minor/patch upgrades by module id.
2. Breaking major upgrades require explicit admin migration action.

---

## 17) Module Import and Registration Process
Required flow:
1. Accept module package/scaffold with manifest.
2. Parse and validate manifest.
3. Persist to `module_registry`.
4. Register installable artifact metadata.
5. For target workspace, create/update `module_installations` record.
6. Wire route and sidebar metadata.
7. Bind declared permissions and access checks.
8. Register gateway target mapping.
9. Activate module in workspace.
10. Support disable and uninstall flows.

Minimum MVP admin UI:
1. Import module package.
2. View validation errors.
3. Install to workspace.
4. Enable/disable in workspace.
5. Uninstall from workspace.
6. View module status and runtime target health.

## 18) Event Bus and Inter-Module Communication
Event bus requirements:
1. Standard event envelope schema.
2. Event publishing API with auth/workspace scope.
3. Event subscription registration by module contract.
4. Durable event storage.
5. Delivery status tracking.
6. Retry strategy for transient failures.
7. Dead-letter handling for repeated failures.
8. Auditability and traceability per event.

Envelope minimum fields:
1. `event_id`
2. `workspace_id`
3. `emitter_module_id` or `shell`
4. `event_name`
5. `payload`
6. `occurred_at`
7. `correlation_id`
8. `delivery_state`

Authorization:
1. Module can emit only declared events.
2. Module can subscribe only declared events.
3. Cross-workspace event leaks are blocked.

## 19) API Gateway Requirements
Gateway requirements:
1. Dynamic route-to-module target mapping from registry/installations.
2. Pre-forward checks: auth, workspace, permission, billing, module-enabled.
3. Forward user/workspace context headers.
4. Enforce module identity boundaries.
5. Normalize and return errors in shell format.
6. Request/response audit logging.
7. Circuit-breaker behavior for unhealthy targets.

Forwarded context minimum:
1. `x-user-id`
2. `x-workspace-id`
3. `x-module-id`
4. `x-permissions` or signed claim equivalent
5. `x-correlation-id`

Impersonation safety:
1. Gateway must include true actor and effective actor metadata where applicable.
2. Impersonation must be explicit in forwarded claims.

---

## 20) Billing and Activation Rules
Billing model responsibilities:
1. Stripe manages payment processing, customer objects, subscription states.
2. Shell manages entitlement enforcement based on billing state.
3. Shell maps workspace ownership to billing account and subscription.

Required shell behaviors:
1. Create/link agency billing account.
2. Track subscription state.
3. Gate module activation when billing invalid.
4. Grace/suspension behavior per policy.
5. Recovery route access for billing update even when suspended.
6. Persist billing enforcement state in shell data model.

MVP enforcement states:
1. `active`: full access according to permissions.
2. `past_due`: warning + configurable grace behavior.
3. `suspended`: block module access, allow recovery/billing routes.
4. `canceled`: restricted access until reactivation.

## 23) Acceptance Criteria (Pass/Fail)
1. Super can create an Agency workspace.
2. Agency can create a Business child workspace.
3. Membership and role assignment controls route/feature/action access correctly.
4. Workspace switching works for multi-workspace users.
5. Super impersonation works and is auditable.
6. Branding/token updates apply consistently across shell screens.
7. Layout drafts can be saved, previewed, tested, published, and rolled back.
8. Manifest-compliant module can be imported and registered without shell source edits.
9. Module routes and sidebar entries appear dynamically when installed/enabled.
10. Module disable/uninstall removes access/routes correctly.
11. Event emission and subscription function through shared event system.
12. Gateway forwards context and blocks unauthorized/disabled access.
13. Billing state can suspend module access while preserving recovery routes.
14. Cross-workspace data isolation holds under API and UI checks.
15. No module domain logic exists in shell codebase.

## 25) Implementation Guardrails for Codex
1. Do not merge shell and module business logic.
2. Do not fake multi-tenancy for later retrofit.
3. Do not hardcode module routes/pages as permanent shell features.
4. Do not bypass shared UI/theme systems with ad hoc styling.
5. Do not make layout editing one-off per page if reusable subsystem is required.
6. Do not skip billing checks in access/activation paths.
7. Do not build speculative future module behaviors into shell.
8. Prefer simple, explicit contracts over clever abstractions.
9. Preserve shell/module boundary whenever uncertain.
10. Treat this PRD and Layout Manager PRD as build governance, not suggestion.
